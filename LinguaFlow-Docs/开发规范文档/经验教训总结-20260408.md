## 🔍 根本原因分析（5 Why 法）

**问题1：文档中心长期不可用**

- **Why 1：** `docs/` 目录从未上传到 GitHub
- **Why 2：** 不知道 GitHub Pages 读取的是 `/docs/` 还是 `/docs-site/`
- **Why 3：** 没有文档维护规范，没有人在变更前确认读取位置
- **Why 4：** Agent 之间没有交接文档维护说明
- **Why 5：** ❌ 缺少《仓库维护手册》，没有明确文档中心的技术架构

**问题2：同一文件出现两个版本**

- **Why 1：** 根目录 `产品文档/` 和 `LinguaFlow-Docs/产品文档/` 同时存在
- **Why 2：** 工程师和 Agent 分别建立了各自的文档目录
- **Why 3：** 没有统一约定文档统一存放位置
- **Why 4：** ❌ 缺少《文档管理规范》

**问题3：index.html 无法推送**

- **Why 1：** `.gitignore` 中有 `index.html`，导致 `docs/index.html` 被忽略
- **Why 2：** 没有人知道这条忽略规则的存在
- **Why 3：** 没有人在变更前检查 `.gitignore` 内容
- **Why 4：** ❌ 缺少《git 变更前检查清单》

## 📊 教训总结（3 条核心）

### 教训 1：文档即代码，变更需测试

文档中心的可用性直接依赖 GitHub Pages 的正确配置。每次对文档目录的任何变更，都必须：
1. 变更前确认 GitHub Pages 读取位置（`/docs/`）
2. 变更后访问文档中心 URL 验证
3. 在 HEARTBEAT.md 中固化此流程

### 教训 2：文档结构必须在变更前确定

随意建立新目录会导致重复。这次事件根源之一是「两个产品文档并存」，造成团队困惑：
- 之后任何新建文档目录，必须在变更前明确「唯一真实来源」
- 不得在两个不同位置维护同一套文档

### 教训 3：Agent 的每一步操作都需要有据可查

Agent 每次 git push 前，必须完成 Regression Test，并在 commit message 中注明测试结果。
