### 第2章 · 用户分析

**2.1 目标用户画像**  
_（都市上班族，25-35岁，英语有进阶需求，碎片化时间多）_

- **画像一**：*通勤者小李* —— 每天地铁通勤40分钟，想利用碎片时间背单词但坚持不下来  
- **画像二**：*职场人王姐* —— 想提升职场英语，但传统背单词 App 太枯燥  

**2.2 核心痛点**  
| 痛点 | 现有解决方案的不足 |
|------|------------------|
| 背单词太枯燥，坚持不过7天 | 传统 App 纯记忆，缺乏情感连接 |
| 记住了单词但不会用 | 缺乏真实语境，学了忘、忘了学 |
| 碎片时间无法专注长内容 | 现有产品单元太重，不适合3-5分钟场景 |
| 单词之间没有关联，记忆孤岛 | 词根词缀学习枯燥，形近词混淆 |

**2.3 典型使用场景**  
1. **地铁通勤**（5-15分钟）：刷一个场景故事 + 记住2-3个新词  
2. **午休碎片**（3-5分钟）：快速复习已学单词  
3. **睡前15分钟**：沉浸式读完一个情感小故事

### 第3章 · Sprint 1 MVP 功能详细描述

> **Sprint 1 周期**：第3~4周（2026-04-20 ~ 2026-05-03）  
> **MVP 目标**：核心学习闭环跑通，上线可用的最小功能集

#### 3.1 用户账号体系
- 手机号 + 验证码登录
- 基础信息管理（昵称、学习目标设置）

#### 3.2 AI 场景故事学习模块
- **故事推荐引擎**：根据用户兴趣标签，推送情感叙事小故事（300-800字）
- **智能生词高亮**：故事内自动识别并高亮目标单词，显示释义
- **情感叙事 Prompt 框架**：AI 生成的故事需满足"有情感共鸣、有代入感、有现实映射"
  - _详细内容占位：Prompt 模板、示例故事、评测标准，见 AI Prompt v1.0 文档_

#### 3.3 单词本与记忆系统
- **个人生词本**：用户收藏的单词，自动关联来源故事
- **艾宾浩斯复习提醒**：根据遗忘曲线智能安排复习时间
- **单词详情页**：音标/美式&英式发音/例句/词根词缀/关联记忆

#### 3.4 学习数据仪表盘
- 今日学习进度（故事数、单词数、复习完成率）
- 学习日历（连续打卡天数）
- 单词掌握度雷达图

### 第4章 · 产品流程（用户路径图）

#### 路径一：首次启动 → 完整体验

```
[打开 App] → [欢迎页 + 目标设置] → [兴趣标签选择]

```

#### 路径三：搜索与探索

```
[点击搜索图标] → [输入单词/话题] 
→ [AI 推荐相关故事] → [阅读故事]
```

### 第6章 · 非功能需求

#### 6.1 性能要求
| 指标 | 要求 |
|------|------|
| App 冷启动时间 | ≤ 3秒（主流机型） |
| 故事内容加载时间 | ≤ 2秒（3G 网络下） |
| API 响应时间（P99） | ≤ 500ms |
| 并发支持用户数 | 初期支持 10万 DAU |

#### 6.2 安全要求
- 用户手机号脱敏存储，不明文暴露
- AI 生成内容须经过基础安全过滤（涉黄涉暴涉政）
- HTTPS 全站加密传输
- 防爬虫与接口滥用保护

#### 6.3 兼容性要求
| 平台 | 最低版本 |
|------|---------|
| iOS | 14.0+ |
| Android | API 24（Android 7.0）+ |
| H5/小程序 | 微信 8.0+、企业微信 4.0+ |

#### 6.4 可用性要求
- 服务可用性目标：99.5%（月度）
- 数据备份：每日增量备份，每周全量备份
- 灾备恢复时间目标（RTO）：≤ 4小时

#### 6.5 技术约束
- 前端：Flutter（iOS + Android 跨平台）
- 后端：Go + FastAPI 双技术栈（各自服务独立部署）
- 数据库：PostgreSQL（主数据）+ Redis（缓存）
- AI 服务：MiniMax API（故事生成、单词解析）
- 部署：Kubernetes（阿里云 ACK）+ GitHub Actions + ArgoCD
